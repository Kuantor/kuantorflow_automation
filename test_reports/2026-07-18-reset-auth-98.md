# Test Report — Reset Auth button

**PR:** kuantorflow#117 · kuantorflow_automation#16 · **Issue:** kuantorflow#98 · **Date:** 2026-07-18

## Summary

Added a **Reset Auth** action to the Settings popup that clears the whole session (gate pass + Google identity) and the app's browser storage, landing back on the gate. Verified end-to-end in the browser and with 6 new tests; offline suite 90 → **96 passed**.

## What changed

- `POST /auth/reset` endpoint: `session.clear()` (both the gate pass and the Google identity live in the signed session cookie — no tokens to invalidate), then redirect to the gate.
- Reset Auth button in the Settings popup, as an *action* outside the settings form: `type="button"`, and deliberately **not** disabled by the #102 read-only freeze (anonymous gated visitors reset too).
- A confirmation dialog (same pattern as card deletion); on confirm the JS clears the app's known browser storage — `kf_mykola_widget_state_v1`, `kf_mykola_auth_dismissed` (localStorage), `kf_welcome_seen_<boot-id>` (sessionStorage) — then submits the native POST.
- Settings files are deliberately preserved: signing back in restores preferences.
- README section documenting the flow.

## Automated tests

- **Added:** `tests/test_reset_auth.py` — 6 tests: signed-in reset clears identity + gate pass (session empty, pages gated again, 302 to `/enter`); anonymous gated visitors can reset; `GET /auth/reset` → 405; the settings file survives a reset; the button renders enabled inside the #102 read-only popup with the confirmation dialog present; full round-trip (reset → keyword re-entry opens the site again).
- **Suite result:** `pytest -m "not live"` — **96 passed**, 1 skipped, 6 deselected (was 90 passed).

## Manual & browser verification

Driven against the local dev server at desktop size, signed out (anonymous), which is the stricter case because of the #102 read-only popup:

- Opened Settings → **Reset Auth button present and enabled** despite every settings control being disabled (read-only freeze). → OK
- Planted `kf_mykola_widget_state_v1` and `kf_mykola_auth_dismissed` in localStorage to confirm clearing. → set
- Clicked Reset Auth → **confirmation dialog shown**, settings popup closed, still on the index page (the button did not submit the settings form). → OK
- Confirmed "Yes, reset" → landed on the **gate** (`/enter`); the planted localStorage keys were **gone**. → OK
- Reloading a page → **gated again** (session cleared server-side). → OK
- Console: no errors.

## Result

Pass. All acceptance criteria met: visible Reset Auth button, session invalidated, keyword/identity cache cleared, redirect to the gate — working for both signed-in and anonymous visitors, with settings files preserved.
